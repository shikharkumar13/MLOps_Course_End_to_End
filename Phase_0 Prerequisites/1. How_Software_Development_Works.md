## Part 1 — What Is Code, Really?

Let's start at the very beginning. **Code is a set of instructions written for a computer to follow.** That's the whole idea.

A computer is incredibly fast but completely literal — it does exactly what it's told, in exact order, and nothing more. Code is how we tell it what to do, written in a language it can understand.

### The Recipe Analogy

Think of code like a **recipe**. A recipe is a list of precise instructions:

```
1. Preheat oven to 180°C
2. Mix flour and sugar
3. Add two eggs
4. Bake for 25 minutes
```

A recipe has a few important properties that code shares exactly:
- **Order matters.** You can't bake before mixing.
- **Precision matters.** "Some flour" is too vague; "200 grams of flour" works.
- **It's just text until someone follows it.** A recipe written on paper doesn't cook anything by itself — a cook has to *execute* it.

Code is the same. It's a list of precise, ordered instructions, written as text, that does nothing until something *executes* it. The "cook" who executes code is the computer.

### What Does Code Look Like?

Here's a tiny piece of real code (in Python, the language used throughout this series):

```python
name = "Alex"
print("Hello, " + name)
```

In plain English, this says: "Remember the word 'Alex' under the label `name`. Then display the words 'Hello, ' followed by whatever is stored in `name`." When executed, it shows: `Hello, Alex`.

That's it. Code is just human-written instructions, often surprisingly readable, that tell the computer to do things step by step.

### Programming Languages

Just as humans have many languages (English, Hindi, Spanish), computers can be instructed in many **programming languages** — Python, JavaScript, Java, and hundreds more. Each has its own grammar and style, but they all serve the same purpose: telling a computer what to do.

For machine learning, **Python** is by far the most popular language, which is why this whole series uses it. You don't need to master Python to follow the MLOps articles — but knowing that code is "just instructions in a language" removes most of the mystery.

## Part 2 — What Is a Program, and What Does "Running" It Mean?

A **program** (also called an **application** or **app**) is code that's been written to accomplish a task. Your web browser is a program. A calculator is a program. The script that trains your ML model is a program.

### "Running" = Executing the Instructions

When you **run** a program, you're telling the computer: "Follow these instructions now." The computer reads the code line by line and does what it says. You'll also hear this called **executing** the program. They mean the same thing.

### The Recipe, Cooked

Back to our analogy:
- The **recipe written on paper** = the code, sitting in a file, doing nothing.
- **Cooking the recipe** = running the program. The instructions actually happen.
- The **finished dish** = the result the program produces.

So when a tutorial says "run `train.py`," it means: "take the file of instructions called `train.py` and have the computer follow them now." The result might be a trained model saved to your disk.

### A Key Realization for Beginners

Code doesn't *do* anything just by existing. Writing code and running code are two separate steps:

```
   WRITE                    RUN
┌──────────┐           ┌──────────────┐
│ You type │           │ Computer     │
│ the      │  ───────► │ executes the │  ───────► Result
│ recipe   │           │ recipe       │
│ (a file) │           │ (cooks it)   │
└──────────┘           └──────────────┘
```

This is why you'll constantly do two things: edit a file (write), then run it (execute) to see what happens. Most of development is this loop: write a little, run it, see the result, adjust, run again.

## Part 3 — Files and Folders: Where Everything Lives

Everything on a computer — your code, your data, your photos, your models — is stored in **files**. Files are organized into **folders** (also called **directories**). This sounds obvious, but having a crisp mental model here prevents enormous confusion later.

### The Filing Cabinet Analogy

Think of your computer's storage as a giant **filing cabinet**:
- A **file** is a single document (a piece of paper).
- A **folder/directory** is a drawer or folder that holds documents and other folders.
- Folders can contain folders, which contain more folders — like nested boxes.

### Files Have Names and Extensions

A file has a name and usually an **extension** — the bit after the dot that signals what *kind* of file it is:

| File name | Extension | What it is |
|-----------|-----------|------------|
| `train.py` | `.py` | A Python code file |
| `heart.csv` | `.csv` | A spreadsheet-like data file |
| `model.pkl` | `.pkl` | A saved (trained) model |
| `README.md` | `.md` | A text document (Markdown) |
| `Dockerfile` | *(none)* | Docker's recipe file |

The extension is a label that tells you (and programs) how to treat the file. A `.py` file is meant to be run as Python code; a `.csv` holds data.

### Paths: A File's Address

To find a file, the computer needs its **path** — the route through the folders to reach it. It's like a postal address.

```
heart_disease_project/        ← top folder
├── data/
│   └── heart.csv             ← path: heart_disease_project/data/heart.csv
├── src/
│   └── train.py              ← path: heart_disease_project/src/train.py
└── README.md                 ← path: heart_disease_project/README.md
```

The path `heart_disease_project/data/heart.csv` reads as: "Inside the `heart_disease_project` folder, go into `data`, and find `heart.csv`." The slashes (`/`) separate the folders, like the parts of an address (country / city / street / house).

> You'll meet two kinds of paths later: **absolute** (the full address from the very top of your computer) and **relative** (directions from wherever you currently are). For now, just know a path is how you point to a specific file. P2 covers this hands-on.

### Project Structure

A well-organized project keeps related files in sensible folders — code in `src/`, data in `data/`, models in `models/`. You saw this in every earlier article. This isn't just tidiness; tools like Git and Docker work *with* your folder structure, so keeping it clean makes everything smoother.

## Part 4 — Your Computer vs. a Server (Local vs. Cloud)

This distinction trips up almost every beginner, so let's make it crystal clear.

### Your Computer = "Local"

Your laptop or desktop is your **local** machine. When you write and run code on it, everything happens *locally* — on the machine in front of you. Only you can see it. If you turn it off, whatever was running stops.

The word **"local"** simply means "on my own computer, right here."

### A Server = A Computer That Serves Others

A **server** is also just a computer — but one whose job is to be available all the time and to *serve* things to other computers over the internet. Servers usually live in data centers (huge buildings full of computers) and stay on 24/7.

### The Restaurant Analogy

- **Cooking at home (local):** You make a meal in your own kitchen. Only the people in your house can eat it. When you go to bed, the kitchen closes.
- **A restaurant (server):** A professional kitchen that's open to the public, serving many customers, staying open on a schedule regardless of any one person.

When you build an ML model on your laptop, you're "cooking at home" — it's local, just for you. To let the *world* use your model, you put it on a **server** — like opening a restaurant.

### "The Cloud" Demystified

**The cloud** is a friendly word for "servers that someone else owns and maintains, which you rent over the internet." Companies like Amazon (AWS), Google (GCP), and Microsoft (Azure) own millions of servers and rent slices of them to anyone.

So "deploying to the cloud" just means "putting your program on a rented server somewhere, so it's always on and reachable by others." There's nothing mystical about the cloud — it's just **other people's computers that you rent.**

```
   LOCAL                            CLOUD / SERVER
┌──────────────┐                ┌──────────────────┐
│ Your laptop  │                │ A rented computer│
│              │   internet     │ in a data center │
│ Only you can │ ◄────────────► │ Always on,       │
│ see it       │                │ anyone can reach │
│ Off when you │                │ it               │
│ close it     │                │                  │
└──────────────┘                └──────────────────┘
```

## Part 5 — What "Deploying" Actually Means

Now the big one. You've heard "deploy" constantly in this series. Here's the plain meaning:

> **Deploying means taking something that works on your computer and making it run somewhere others can reliably use it.**

That's the entire concept. You built a model locally; *deploying* it means putting it somewhere — usually a server — where other people or programs can send it data and get predictions back, anytime.

### The Restaurant Analogy, Completed

Think of the full journey of opening a restaurant:

| Cooking journey | Software journey |
|-----------------|------------------|
| Perfect a recipe in your home kitchen | Build and test a model on your laptop |
| Standardize it so any cook can make it | Package it so it runs the same anywhere (**Docker!**) |
| Rent a restaurant space | Rent a server (**the cloud**) |
| Open the doors to the public | **Deploy** — make it reachable by others |
| Take orders at the counter | Receive requests through an **API** (**Flask/FastAPI**) |

Suddenly the whole series clicks into place. Every tool you've learned plays a role in this journey from "works on my laptop" to "anyone can use it":

- **Git** keeps your recipe's history and lets a team cook together.
- **DVC** versions your ingredients (data) and finished dishes (models).
- **Docker** standardizes your kitchen so the dish tastes identical anywhere.
- **Flask/FastAPI** is the counter where customers place orders.
- **The cloud** is the rented restaurant space.
- **Deploying** is opening the doors.

### Why Deployment Is Its Own Skill

A common beginner surprise: a model that works perfectly on your laptop often breaks when deployed. Different computer, missing ingredients, wrong versions — exactly the "it works on my machine" problem Docker solves (Article 4). This is *why* MLOps tools exist: to make the journey from local to deployed smooth and reliable. Deployment isn't an afterthought; it's a real engineering skill, and it's what this whole series teaches.

## Part 6 — The Client-Server Model

One last concept and you'll have the full picture. Almost all modern software follows the **client-server model**, and you've been quietly using it for several articles.

### The Two Roles

- A **server** offers a service and waits for requests. (Your deployed ML model.)
- A **client** is whoever *uses* that service by sending a request. (A web app, a mobile app, another program, or even you testing with a command.)

### The Waiter Analogy (Again, Because It's Perfect)

- The **client** is the customer who places an order.
- The **server** is the kitchen that prepares and returns the dish.
- The **request** is the order ("I want a prediction for this patient").
- The **response** is what comes back ("prediction: heart disease likely").

```
   CLIENT                              SERVER
(asks for something)              (provides it)
      │                                 │
      │   ──── request ────►            │
      │   "predict for this data"       │
      │                                 │ runs the model
      │   ◄─── response ────            │
      │   "prediction: 1"               │
      │                                 │
```

This back-and-forth — request out, response back — is the fundamental rhythm of deployed software. When you built APIs with Flask and FastAPI, you were building the **server** side. The `curl` commands and browser visits were acting as the **client**. Now you know the names for both halves.

## Part 7 — How All of This Fits an ML Project

Let's tie the whole mental model together with the journey of a real ML project — the heart disease classifier you've seen throughout this series.

```
1. WRITE CODE (local)
   You write train.py on your laptop — instructions to train a model.
        │
2. RUN IT (local)
   You execute train.py. The computer follows the recipe and produces
   model.pkl — a saved, trained model file.
        │
3. VERSION IT
   Git tracks your code's history; DVC tracks the data and model files.
        │
4. WRAP IT IN AN API
   You write an API (Flask/FastAPI) — a "counter" where clients can
   send patient data and receive predictions.
        │
5. PACKAGE IT
   Docker bundles your code, the model, and everything it needs into
   one standardized container that runs identically anywhere.
        │
6. DEPLOY IT (local → server/cloud)
   You put the container on a server. Now it's always on and reachable.
        │
7. CLIENTS USE IT
   Web apps, mobile apps, or other programs (clients) send requests to
   your server and get predictions back. Your model is live.
```

Every single step here uses a concept from this article: **code** (step 1), **running** (step 2), **files** (the model artifact), **versioning** (step 3), **client-server** (steps 4 & 7), **local vs. server** (step 6), and **deploying** (step 6). The MLOps tools are just the practical machinery for carrying out this journey reliably.
